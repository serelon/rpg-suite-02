# The thinking block is where a skill gets *enforced*, not just hoped for

**What it is.** Every counter-default rule fights a trained reflex, and **reflexes win under
load unless something interrupts the autopilot before the output commits.** The thinking
block is that interrupt: a deliberate pre-commit check is the mechanism by which intentions
actually beat reflexes. So thinking isn't auxiliary — it's *where the rest of the skill
executes*. It's also the **private back-of-house** where all "zero-trace" machinery lives
(rolls, oracle draws, computed stamps, OOC absorption), keeping the output clean.

Discipline: **heavy but genuine, never rote.** The failure isn't length, it's *hollow
recitation* — running the menu as a checkbox. The trained failure in creative work is to
*under*-think (rush to the fun part), so a soft "consider thinking about X" gets skipped;
the mandate is emphatic on purpose.

**Where it comes from.** [[gm-skill-SKILL]] §"Thinking"; [[gm-skill-RATIONALE]] §"the
enforcement layer" (called the highest-leverage section, added late).

**Why it matters for next-gen.** A general principle: **counter-default behavior needs an
enforcement surface, not just a specification.** Wherever the framework wants behavior that
fights a default, there must be a pre-commit checkpoint that fires it. Completes
[[0013-counter-training-name-the-default]] (explain the rule *and* give it a place to fire)
and is the runtime guard against [[0014-scope-stripping]] / [[0015-compounding-loops]] (the
meta-checks only a thinking pass can do).

**Open threads.** Enforcement is token-expensive every turn — tension with context economy.
Can the check be *adaptive* (deep at boundaries, light mid-scene) without becoming rote?
What's the analogue for a small/local model with a weaker or absent thinking channel?
